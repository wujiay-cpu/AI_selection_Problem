## 算法选择

后端固定使用 `run_backtracking_pruning` 算法（先贪心后回溯精确搜索）。前端不暴露算法选择，所有规模统一走该算法路径，与脚本基线测试一致。

## 前端使用说明

### 复现脚本基线

1. 选择手动模式。
2. `selected_numbers` 输入 `1, 2, ..., n`（与 case 的 `n` 对应）。
3. `k`、`m(=j)`、`t(=s)` 与测试表一致。
4. 提交后等待返回。

### 关于随机模式

随机模式从 `1..m` 中抽取 `n` 个元素作为 pool。由于算法只关心集合关系，不依赖元素数值本身，结果应与基线一致；当 pool 大小或参数不同，结果不同属于正常现象。

### 运行时间预期

- case 01-10、12：约 30s 内完成
- case 11/13/14：约 100-400s（已知限制，build 阶段不可中断）
- case 15（`n=28`）：不支持

前端客户端超时已设为 400 秒，可容纳大 case。

## 偏差分析

### 偏差较大的 case

- case 02（`+41`）：理论下界与目标重合，短时限内接近不可达。
- case 05（`+53`）：候选空间较大，贪心 seed 阶段占满预算。
- case 13（`+13`）：建模与搜索成本高，DFS/ILS 预算受限。

### 时间超限 case

- case 11、13、14：大规模实例在 build/greedy/DFS 阶段耗时高，且存在随机波动。

## 算法验证函数

```python
from itertools import combinations

def verify_cover(pool, k, j, s, result):
    for jsub in combinations(pool, j):
        jset = set(jsub)
        if not any(len(jset & set(b)) >= s for b in result):
            return False
    return True
```
